# 프로젝트 개요 및 개발 방법

> 프로젝트명(가칭): **다이소 상품 위치 안내 RAG 기반 AI 검색 서비스**  
> 운영 목표: **기능별로 분리 개발(서비스/모듈 단위) → 병합(통합 실행) → 로컬 통합 테스트(TDD) 통과 → AWS Lightsail 배포**  
> 추가 요구사항: **병합 시 프로세스 생성 오버헤드(불필요한 프로세스/컨테이너 난립) 최소화**

---

## 1. 프로젝트 개요

### 1.1 문제/가치
- 매장 직원의 반복적인 “상품 위치 안내” 부담을 시스템이 대체하여 **업무 효율(생산성)** 및 **고객 경험**을 개선.
- PoC 흐름(권장): **STT → 의도분석 → 키워드 추출/확장 → Hybrid 검색(BM25+Vector) → 리랭킹 → 위치안내 + QR 인계**

### 1.2 핵심 기능(Functional Scope)
1) **입력**: 음성(스트리밍 STT) / 텍스트  
2) **의도분석 게이트**: 검색 대상(상품 위치 문의)인지 판별 (In/Out-of-scope)  
3) **의도(키워드) 추출/확장**: 검색에 유효한 키워드 후보 생성 및 보강  
4) **하이브리드 검색**: BM25 + Vector(Qdrant) + Fusion(RRF/가중합)  
5) **리랭킹**: Top-K 후보를 모델로 재정렬해 최종 Top1 확정  
6) **출력(UI)**: Top3 상품(Top1 강조) + 위치 문구 + QR(모바일 인계)  
7) **로그/관측성**: 질의-응답, latency, hit@k, 실패/애매함 비율 모니터링

### 1.3 비기능 요구사항(Non-Functional)
- **지연 시간 목표(초기 가이드)**: 키오스크 UX를 고려해 “체감 2초 내외”를 목표(단계별 SLA 분리: STT, 검색, 리랭킹).  
- **신뢰성**: 장애 시 Fallback(텍스트 입력, 캐시된 결과, BM25-only로 축소 운영 등)  
- **보안/비용**: API 키/시크릿 관리, 음성 데이터 보관 정책, 월 비용 상한 설정

## 2. 기술 스택(권장안)

### 2.1 애플리케이션
- 런타임: **Node.js (TypeScript 권장)**
- API: REST(초기) + 필요 시 gRPC(서비스 간)
- 테스트: **Jest**(단위/통합), Supertest(HTTP), Nock(외부 API mocking)
- 데이터:
  - Real DB: PostgreSQL 또는 MySQL (최소 기능부터)
  - Cache/State: Redis (세션/레이트리밋/결과 캐시)
  - Retrieval: **Elasticsearch(BM25)**, **Qdrant(Vector)**
- AI/외부:
  - STT: Google Streaming STT(또는 대안)
  - LLM: 의도분석/키워드 추출/리랭킹(벤더 세트로 교체 가능)
  - Embedding: 벤더 세트 기반 교체(OpenAI/Google 등)

### 2.2 인프라(배포)
- **AWS Lightsail** (MVP/파일럿 친화)
  - 권장: Lightsail **Instances + Docker Compose**
  - 대안: Lightsail **Containers**(운영 편의성↑, 구성 제약/로그/네트워킹 확인 필요)

## 3. 아키텍처 원칙(“MSA 고려” + 현실적 운영)

> 초기부터 완전 MSA(다수 독립 배포 단위)를 강제하면 운영 비용이 급증합니다.  
> 권장: **“논리적 MSA(코드/경계) → 통합 실행(병합) → 필요 시 물리적 MSA(독립 배포)”** 단계적 전환

### 3.1 논리적 서비스 분해(권장)
- **api-gateway**: 단일 인바운드, 인증/레이트리밋, 라우팅
- **stt-service**: 음성 스트리밍 → 텍스트, 품질 게이트
- **nlu-service**: 의도분석(In/Out) + 키워드 추출/확장
- **retrieval-service**: BM25(Elastic) + Vector(Qdrant) + Fusion
- **rerank-service**: 후보 재정렬, Top1 확정
- **location-service**: 상품/카테고리 → 매장 위치 매핑(문구/좌표)
- **handover-service**: QR 생성, 모바일 인계 세션
- **telemetry(공통)**: 지표/로그/트레이싱

### 3.2 데이터/상태 레이어
- Real DB: 상품/카테고리/매장 좌표/결과 이력/세션  
- Redis: 캐시(검색/리랭킹), 레이트리밋 토큰버킷, 세션 상태  
- 검색 인덱스: Elastic/Qdrant (인덱스 생성/갱신 전략 포함)

## 4. 개발 방법론: **Critical — TDD 강제**

### 4.1 TDD 절대 규칙
- **테스트 없이 구현 코드 먼저 작성 금지**
- **실패(Red) 확인 없이 구현 금지**
- 구현 중 **테스트 코드 수정 금지**(테스트 통과 후 리팩토링 단계에서만 허용)

### 4.2 TDD 개발 순서(고정)
1) 테스트 코드 작성(성공 기준을 코드로 고정)  
2) 테스트 실행 → 실패 확인(Red)  
3) 최소 구현(Green)  
4) 전체 테스트 재실행 → 통과 확인  
5) 리팩토링(모듈화/중복 제거/경계 강화)  
6) 회귀 테스트 재실행(항상 Green 유지)

### 4.3 테스트 범위 정의
- **단위(Unit)**: 키워드 추출, fusion scoring, 룰 기반 게이트 등
- **통합(Integration)**: HTTP 핸들러 + DB/Redis(테스트 컨테이너/모킹)
- **계약(Contract, 선택)**: gateway ↔ 서비스 응답 스키마 고정(JSON)
- **E2E(선택)**: “텍스트/음성 → Top3 payload” 전체 흐름

### 4.4 품질 기준
- 테스트 커버리지: **80% 이상**(핵심: nlu/retrieval/rerank는 더 높게 권장)
- 정적 분석: ESLint + TypeScript strict(권장)
- 포맷: Prettier(권장)
- CI: PR마다 테스트/커버리지/린트 실행

## 5. 개발/병합/배포 프로세스(현재 운영 방식 반영)

> 현재 상태: **기능별로 각각 개발 완료(서비스/모듈 단위)**  
> 목표: **병합해서 통합 실행 → 로컬에서 테스트 전부 통과 → Lightsail 배포**

### 5.1 브랜치/병합 전략(권장: 단순/안전)
- `main`: 항상 배포 가능한 상태(테스트 Green 유지)
- `feature/*`: 기능 단위 개발 브랜치
- `integration`: (선택) 병합/충돌 해결 및 통합 테스트 전용 브랜치
- 병합 규칙:
  - feature 브랜치에서 **단위 테스트 Green**
  - integration에서 **통합 테스트 Green**
  - main으로 머지 후 **전체 테스트 Green** 확인

### 5.2 병합 후 “통합 실행” 체크리스트
- 공통 타입/DTO 스키마(`shared-types`) 충돌 여부
- 서비스 간 엔드포인트/포트/환경변수 키 이름 일치
- 로컬 실행용 `docker-compose.local.yml` 또는 `.env.local` 정합성
- 외부 의존성(Elastic/Qdrant/Redis/DB) 헬스체크 통과
- e2e 최소 시나리오 3개 이상 통과(예: 욕실매트/건전지/잡음발화)

### 5.3 로컬 테스트 “게이트” (배포 전 필수)
- **단위 테스트 전체 통과**
- **통합 테스트 전체 통과**
- (선택) 커버리지 80% 이상 충족
- (선택) 린트/타입체크 통과

> 배포 금지 조건(하나라도 해당 시):  
> - 테스트 미통과, 테스트 누락, 테스트를 수정해서 억지 통과, 환경변수 불일치, 계약 스키마 불일치

### 5.4 병합 시 “프로세스 생성 오버헤드” 최소화 원칙(추가 요구사항)

> 병합 후 통합 실행 단계에서 **불필요한 프로세스/컨테이너 난립**이 생기면, 로컬 테스트 속도·운영비·장애점이 급격히 증가합니다.  
> 따라서 “논리적 MSA 유지 + 물리적 프로세스 수 최소화”를 병합 기준으로 고정합니다.

- **원칙 A — 통합 실행은 최소 프로세스 세트로**
  - 로컬/파일럿 단계에서는 `api-gateway` 중심의 **단일 런타임(또는 최소 2~3개)** 로 합쳐 실행(모듈은 내부 패키지로 유지).
  - 즉, 코드 경계는 분리(논리적 MSA)하되, 실행 프로세스는 합치기(모놀리식 런타임)로 오버헤드를 억제.

- **원칙 B — “기능별 서비스”는 병합 시 모듈로 링크**
  - 서비스 간 통신을 HTTP로만 강제하지 않고, 병합 모드에서는 **in-process 호출(모듈 import)** 로 전환 가능하도록 어댑터 계층을 둠.
  - 예: `retrieval-client`가 실행 모드에 따라 `httpClient` 또는 `inProcessAdapter`를 선택.

- **원칙 C — 중복 워커/스케줄러 금지**
  - 큐/배치/스케줄링(예: cron)은 병합 모드에서 **단일 워커**만 실행되도록 락(리더 선출/redis 락)을 둠.
  - “같은 작업을 여러 프로세스가 중복 실행”하는 형태를 금지.

- **원칙 D — Compose 프로파일로 실행 토폴로지 고정**
  - `docker-compose`의 profile(예: `merged`, `msa`)로 실행 형태를 고정:
    - `merged`: 최소 프로세스(통합 런타임) + 외부 의존성(Elastic/Qdrant/Redis/DB)
    - `msa`: 서비스별 분리 프로세스(확장/성능 테스트 용도)
  - 병합/로컬 테스트는 기본적으로 `merged` 프로파일만 허용.

- **원칙 E — Node 프로세스/포트 난립 방지**
  - 개발 서버(핫리로드)도 병합 모드에서는 **단일 dev 서버**를 사용(필요 시 워크스페이스 번들링).
  - 포트는 `api-gateway` 1개만 외부 노출, 내부는 모듈화로 해결.

- **검증(필수 체크)**
  - 병합 후 로컬 통합 실행 시, 프로세스 수/컨테이너 수가 목표 범위를 초과하면 배포 금지.
  - 예: `merged` 기준 “애플리케이션 런타임 1~3개 + 인프라(Elastic/Qdrant/Redis/DB)” 정도로 유지.

## 8. 배포(Deploy) — AWS Lightsail 기준

### 8.1 배포 원칙(현재 프로세스 반영)
- **로컬에서 통합 실행 + 전체 테스트 통과(Green) 확인 후에만 배포**
- 배포는 `api-gateway`만 외부 노출(나머지는 내부 네트워크)
- 장애/비용을 고려해 초기엔 단일 인스턴스에 구성 가능하나, **DB/검색/캐시는 추후 분리 전제**로 설계

### 8.2 MVP 배포 전략(권장)
- Lightsail Instance 1대(초기)  
- Nginx(Reverse Proxy) + Docker Compose
- 내부 서비스는 docker network로 통신
- Elastic/Qdrant/Redis/DB:
  - 초기: 같은 인스턴스(비권장 장기, 단 MVP 가능)
  - 확장: 검색/DB를 별도 인스턴스 또는 관리형으로 분리

### 8.3 환경변수/시크릿
- `.env` 커밋 금지
- 키(LLM/STT/Elastic/Qdrant)는 Lightsail 환경변수 또는 Secrets Manager(확장 시)

## 9. Claude Code / LLM 기반 개발 운영 규칙(요약)

핵심은 “검증 가능한 성공 기준”을 항상 제공하는 것입니다(테스트/명령/기대 결과). 또한 **탐색→계획→구현**을 분리하면 잘못된 방향으로 코딩하는 비용을 크게 줄입니다. 컨텍스트가 길어지면 품질이 떨어지므로 작업 단위를 쪼개고, 병합 단계에서는 **통합 테스트 결과(로그/실패 케이스)**를 우선 첨부하는 방식이 효율적입니다.

### 9.1 이 프로젝트용 작업 지시 템플릿(복사/사용)
```
목표: (한 줄로)
성공기준: (Jest 테스트가 Red→Green, 테스트 수정 금지)
범위: (수정 가능한 폴더/파일)
금지: 구현 먼저 작성 금지. 반드시 테스트 → 실패 확인 → 구현.
명령: (실행할 테스트/빌드 명령)
```

## 10. 마일스톤(현실적 순서)

1) **M0: 병합/통합 실행 안정화**
- 기능별 구현을 병합하고, 통합 실행 docker-compose.local(merged profile)로 고정
- 로컬에서 “최소 e2e 3케이스” Green
- 병합 모드 프로세스 수 목표: **애플리케이션 런타임 1~3개**

2) **M1: 하이브리드 검색 지표 고정**
- hit@k/mrr/ndcg를 매 릴리즈마다 비교 가능하도록 템플릿화

3) **M2: 리랭킹/애매함 처리 고도화**
